    29→## 1. Product Definition & Scope
    30→
    31→### 1.1 What This Product Is
    32→
    33→Sovereign is a browser-based web application that turns real-life self-improvement into an RPG kingdom-building game. Users complete real-world actions — studying, exercising, building skills, leading — and those actions award XP, level up their character, grow attributes, and unlock buildings in a persistent kingdom.
    34→
    35→It is NOT a to-do list. It is NOT a habit tracker with cosmetic decoration. It is an identity system that happens to use quests as its input mechanism.
    36→
    37→### 1.2 What This Product Is Not
    38→
    39→- Not a downloadable game (no Unity, Godot, or game engine)
    40→- Not a mobile app (no React Native or Flutter)
    41→- Not a social media platform
    42→- Not a habit tracker
    43→- Not a task manager
    44→- Not a dashboard or analytics tool
    45→
    46→The kingdom map is a styled CSS div layout. The XP bar is a CSS transition. The AI companion is a fetch call. Everything runs in a browser via URL.
    47→
    48→### 1.3 Target Platform
    49→
    50→**FIXED:** Next.js 14 web application, deployed on Vercel, accessed via browser at a custom domain.
    51→
    52→Same technical form factor as Linear, Notion, and Habitica — all of which are React/Next.js web apps with game-like UI.
    53→
    54→### 1.4 Target User
    55→
    56→Primary demographic: Ambitious males aged 18–24 who consume self-improvement content (YouTube, Reddit, podcasts), are familiar with gaming progression systems (anime leveling aesthetics, RPG stat screens), and feel productivity apps are either too boring or too childish.
    57→
    58→**IMPORTANT — This demographic is a hypothesis, not a fact.** The 20-person test cohort in Phase 2 Week 14 must validate this assumption. The product should be built as though the assumption is true, but the retention metrics will reveal whether it holds.
    59→
    60→Evidence supporting the hypothesis:
    61→- HabitForge (Solo Leveling-inspired habit tracker) gained a user base in this demographic with anime-leveling aesthetics
    62→- Reddit communities such as r/selfimprovement, r/nosurf, and r/getdisciplined skew young male
    63→- LifeUp has 280K+ downloads with user reviews indicating gaming-oriented self-improvement seekers
    64→
    65→Evidence against: LifeUp's top complaint is that it "lacks a real game." Habitica lost its community spaces in 2023 and its guilds in 2023 due to moderation issues. This means the category has a demand problem (users want a more serious experience) and a supply gap.
    66→
    67→### 1.5 Core Loop
    68→
    69→**Single session (under 5 minutes for casual engagement):**
    70→
    71→```
    72→Open app
    73→→ Check active quests on Dashboard
    74→→ Mark objective(s) complete on one quest
    75→→ See XP animation fill the bar
    76→→ [Optional] Check kingdom for any building unlocks
    77→→ [Optional] Message companion
    78→→ Close app
    79→```
    80→
    81→**Daily loop (5–15 minutes for power users):**
    82→
    83→```
    84→Morning: Check companion for daily quest recommendation
    85→→ Browse Quest Board for available quests
    86→→ Start 1–2 new quests
    87→→ Mark progress on active quests
    88→
    89→Evening: Mark completions
    90→→ XP award and animation
    91→→ Level-up gate if threshold crossed
    92→→ Building unlock check
    93→→ Achievement check
    94→→ Check guild weekly progress
    95→```
    96→
    97→**Weekly loop:**
    98→
    99→```
   100→Monday: House weekly collective XP goal resets
   101→→ New featured quest appears on Quest Board
   102→→ Review progress toward Kingdom Level milestone
   103→
   104→Friday: Check House leaderboard position
   105→→ Final push on active quests before week resets
   106→```
   107→
   108→### 1.6 Session Length Targets
   109→
   110→- Minimum meaningful session: 30 seconds (mark one objective complete)
   111→- Casual session: 2–5 minutes (check dashboard, update quest, view kingdom)
   112→- Power user session: 10–20 minutes (quest board review, companion chat, character review)
   113→- Onboarding session: 90 seconds maximum (3 steps, one page)
   114→
   115→### 1.7 Win State and Endgame
   116→
   117→There is no win condition. Sovereign is an ongoing identity game.
   118→
   119→Milestones that replace a "win state":
   120→- Reach Level 10 (Journeyman status)
   121→- Build all 6 Phase 1 kingdom buildings
   122→- Complete quests in every domain (Renaissance achievement)
   123→- Reach Kingdom Level 5 (Phase 2)
   124→- Reach the top 10 of the House weekly leaderboard (Phase 2)
   125→
   126→The "endgame" is a high-level character with a complete kingdom, a recognizable title in the leaderboard, and a reputation in their House. Phase 3 will introduce prestige mechanics (class reset with bonuses) for users who have reached Level 50+.

   130→## 2. Technical Architecture
   131→
   132→### 2.1 Framework and Engine
   133→
   134→**FIXED: Next.js 14 with App Router. No other framework is acceptable.**
   135→
   136→Justification:
   137→- Single repo for frontend, API routes, and server components
   138→- App Router enables server-side data fetching without separate API layer for most reads
   139→- Vercel deploys Next.js natively with zero configuration
   140→- shadcn/ui is built for Next.js + Tailwind
   141→- Supabase has a first-class Next.js integration package
   142→
   143→Do NOT use: Vite+React (no SSR, no API routes), Remix (less ecosystem support with Supabase), SvelteKit (unfamiliar tech stack risk).
   144→
   145→### 2.2 Frontend Stack
   146→
   147→| Layer | Technology | Version | Purpose |
   148→|---|---|---|---|
   149→| Framework | Next.js | 14.2.x | App Router, SSR, API Routes |
   150→| Language | TypeScript | 5.x | Type safety |
   151→| Styling | Tailwind CSS | 3.4.x | Utility-first CSS |
   152→| Components | shadcn/ui | latest | Accessible primitives: Button, Dialog, Input, Tabs, Badge, Sonner |
   153→| Fonts | next/font/google | built-in | Cormorant Garamond, Space Grotesk, Space Mono |
   154→| Animations | CSS transitions only (Phase 1) | — | 90% of animations need no library |
   155→| Icons | Lucide React | latest | Bundled with shadcn; used for UI only, not kingdom icons |
   156→| State | React Context + useState | built-in | No external state library at Phase 1 |
   157→| Form handling | React Hook Form + Zod | latest | Onboarding and settings forms only |
   158→
   159→**No Redux. No Zustand. No Jotai.** React Context is sufficient for Phase 1 auth state and user profile.
   160→
   161→### 2.3 Backend / BaaS
   162→
   163→**FIXED: Supabase only. Do not split across multiple services.**
   164→
   165→| Supabase Feature | Used For | Free Limit | Phase |
   166→|---|---|---|---|
   167→| Auth | Email+password login, Google OAuth, session tokens | 50,000 MAU | 1 |
   168→| PostgreSQL | All data: profiles, quests, buildings, messages | 500MB | 1 |
   169→| Row Level Security | Users read only their own rows | Built-in | 1 |
   170→| Edge Functions | AI companion calls, quest generation (server-side Gemini calls) | 500K/month | 1 |
   171→| Realtime | House/guild activity feed | 200 concurrent | 2 |
   172→| Storage | Avatar uploads (avoid — use DiceBear instead) | 1GB | Never |
   173→
   174→**Critical operational note:** Supabase free tier pauses projects after 7 consecutive days of zero database requests. To prevent this during development: set up a GitHub Actions scheduled workflow to ping the health endpoint every 3 days. Do NOT let the project pause — data is preserved but the downtime breaks demos.
   175→

   177→```yaml
   178→name: Supabase Keepalive
   179→on:
   180→  schedule:
   181→    - cron: '0 12 */3 * *'  # Every 3 days at noon UTC
   182→jobs:
   183→  ping:
   184→    runs-on: ubuntu-latest
   185→    steps:
   186→      - run: curl -s "${{ secrets.SUPABASE_URL }}/rest/v1/" -H "apikey: ${{ secrets.SUPABASE_ANON_KEY }}" > /dev/null
   187→```

   189→### 2.4 AI Integration
   190→
   191→**FIXED: Gemini 3 Flash as primary model. Gemini 2.0 Flash was deprecated and shut down June 1, 2026 — do not reference it anywhere in code.**
   192→
   193→| Service | Model | Free Limit | RPM | RPD | Use |
   194→|---|---|---|---|---|---|
   195→| Google AI Studio | gemini-3-flash (or `gemini-2.5-flash` if 3 unavailable) | Free, no credit card | 10 | 1,500 | Companion chat, quest lore personalization |
   196→| Groq | llama-3.1-8b-instant | Free, no credit card | varies | 14,400 | Fallback when Gemini unavailable |
   197→
   198→**API key management:**
   199→- `GEMINI_API_KEY` — server-side only, in `.env.local`, never in client code
   200→- Never prefix with `NEXT_PUBLIC_` — would expose the key in browser
   201→- All Gemini calls go through Next.js API Route Handlers in `/app/api/`
   202→
   203→**AI strategy:** Template-based + light AI personalization. The AI does NOT generate quest content. It only:
   204→1. Personalizes lore text for existing quest templates (1 API call per quest generation, ~20 words output)
   205→2. Responds to companion chat messages (1 API call per message, 200 token max output)
   206→
   207→At 50 DAU × 3 AI calls/day = 150 calls/day. Well within 1,500 RPD free limit.
   208→
   209→**Rate limit handling:**
   210→```typescript
   211→// lib/gemini.ts — exponential backoff on 429
   212→async function callGemini(prompt: string, maxRetries = 3): Promise<string> {
   213→  for (let attempt = 0; attempt < maxRetries; attempt++) {
   214→    try {
   215→      const response = await fetch(GEMINI_API_URL, { ... })
   216→      if (response.status === 429) {
   217→        await sleep(Math.pow(2, attempt) * 1000) // 1s, 2s, 4s
   218→        continue
   219→      }
   220→      return await parseResponse(response)
   221→    } catch (e) {
   222→      if (attempt === maxRetries - 1) throw e
   223→    }
   224→  }
   225→  return RULE_BASED_FALLBACK // If all retries fail, use rule-based response
   226→}
   227→```
   228→
   229→**Fallback to Groq:** If Gemini returns 429 after 3 retries, call Groq's llama-3.1-8b-instant with the same prompt. If Groq also fails, return a rule-based response from the RULE_RESPONSES dictionary (no API call).

   261→### 2.6 Asset Pipeline
   262→
   263→**Kingdom building icons:**
   264→- Source: Game-Icons.net (CC-BY license — attribution required in footer)
   265→- Format: SVG, inline embedded or fetched as static files in `/public/icons/`
   266→- Rendering: CSS `filter: brightness(0) invert(0.9)` for warm white on dark background
   267→- Do NOT use `<img>` tags for SVG icons — use inline `<svg>` or CSS background-image for better control
   268→
   269→**Character avatars:**
   270→- Source: DiceBear API — `https://api.dicebear.com/9.x/adventurer/svg?seed={userId}&backgroundColor=080808`
   271→- No uploads, no storage costs, consistent per user ID
   272→- Fallback: First character of character name if DiceBear unreachable
   273→
   274→**Fonts:**
   275→- All loaded via `next/font/google` in `app/layout.tsx` — no external HTTP request at runtime
   276→- Cormorant Garamond: weights 400, 700 (normal and italic)
   277→- Space Grotesk: weights 400, 500, 600
   278→- Space Mono: weights 400, 700

   282→## 3. Complete Data Model
   283→
   284→### 3.1 Database Tables — Phase 1
   285→
   286→All tables are in the `public` schema. All tables have Row Level Security enabled.
   287→
   288→```sql
   289→-- ============================================================
   290→-- CORE USER DATA
   291→-- ============================================================
   292→
   293→-- User profiles (extends Supabase auth.users)
   294→CREATE TABLE public.profiles (
   295→  id                  UUID REFERENCES auth.users(id) ON DELETE CASCADE PRIMARY KEY,
   296→  character_name      TEXT NOT NULL CHECK (char_length(character_name) BETWEEN 2 AND 30),
   297→  kingdom_name        TEXT NOT NULL CHECK (char_length(kingdom_name) BETWEEN 2 AND 40),
   298→  companion_name      TEXT NOT NULL DEFAULT 'Aegis' CHECK (char_length(companion_name) BETWEEN 2 AND 20),
   299→  class               TEXT NOT NULL CHECK (class IN ('scholar', 'warrior', 'builder', 'commander')),
   300→  house_id            TEXT NOT NULL DEFAULT 'zenith' CHECK (house_id IN ('ash', 'zenith', 'forge', 'crown')),
   301→  house_changed_at    TIMESTAMPTZ,                  -- Enforce once-per-week change limit
   302→  level               INTEGER NOT NULL DEFAULT 1 CHECK (level >= 1),
   303→  xp                  INTEGER NOT NULL DEFAULT 0 CHECK (xp >= 0),
   304→  xp_to_next          INTEGER NOT NULL DEFAULT 300, -- XP needed for current level → next level
   305→  xp_total            INTEGER NOT NULL DEFAULT 0,   -- Lifetime XP (never resets on level-up)
   306→  kingdom_level       INTEGER NOT NULL DEFAULT 1 CHECK (kingdom_level >= 1),
   307→  kingdom_xp          INTEGER NOT NULL DEFAULT 0,   -- Separate from character XP
   308→  -- Attributes (all start at 10, grow from quest completion)
   309→  attr_strength       INTEGER NOT NULL DEFAULT 10 CHECK (attr_strength >= 0),
   310→  attr_vitality       INTEGER NOT NULL DEFAULT 10 CHECK (attr_vitality >= 0),
   311→  attr_intelligence   INTEGER NOT NULL DEFAULT 10 CHECK (attr_intelligence >= 0),
   312→  attr_focus          INTEGER NOT NULL DEFAULT 10 CHECK (attr_focus >= 0),
   313→  attr_technical      INTEGER NOT NULL DEFAULT 10 CHECK (attr_technical >= 0),
   314→  attr_creativity     INTEGER NOT NULL DEFAULT 10 CHECK (attr_creativity >= 0),
   315→  attr_leadership     INTEGER NOT NULL DEFAULT 10 CHECK (attr_leadership >= 0),
   316→  attr_charisma       INTEGER NOT NULL DEFAULT 10 CHECK (attr_charisma >= 0),
   317→  attr_discipline     INTEGER NOT NULL DEFAULT 10 CHECK (attr_discipline >= 0), -- Streak-based
   318→  -- Meta
   319→  onboarding_complete BOOLEAN NOT NULL DEFAULT FALSE,
   320→  created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
   321→  updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
   322→);
   323→
   324→-- Trigger: auto-update updated_at
   325→CREATE OR REPLACE FUNCTION update_updated_at()
   326→RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = NOW(); RETURN NEW; END; $$ LANGUAGE plpgsql;
   327→CREATE TRIGGER profiles_updated_at BEFORE UPDATE ON public.profiles
   328→  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
   329→
   330→-- ============================================================
   331→-- QUEST SYSTEM
   332→-- ============================================================
   333→
   334→-- Quest templates (seeded by developer, not user-generated at Phase 1)
   335→CREATE TABLE public.quest_templates (
   336→  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   337→  title               TEXT NOT NULL CHECK (char_length(title) BETWEEN 5 AND 80),
   338→  description         TEXT NOT NULL CHECK (char_length(description) BETWEEN 10 AND 200),
   339→  domain              TEXT NOT NULL CHECK (domain IN ('body', 'mind', 'craft', 'command')),
   340→  difficulty          TEXT NOT NULL DEFAULT 'medium' CHECK (difficulty IN ('easy', 'medium', 'hard', 'epic')),
   341→  class_affinity      TEXT[] DEFAULT '{}',           -- Empty array = available to all classes
   342→  xp_reward           INTEGER NOT NULL CHECK (xp_reward > 0),
   343→  rarity              TEXT NOT NULL DEFAULT 'common' CHECK (rarity IN ('common', 'uncommon', 'rare', 'epic')),
   344→  duration_days       INTEGER NOT NULL DEFAULT 7,
   345→  objectives          JSONB NOT NULL DEFAULT '[]',   -- [{id: UUID, text: string, order: number}]
   346→  -- Attribute rewards on completion
   347→  primary_attr        TEXT NOT NULL,                 -- Which attribute gets +2
   348→  secondary_attr      TEXT NOT NULL,                 -- Which attribute gets +1
   349→  kingdom_xp_reward   INTEGER NOT NULL DEFAULT 50,   -- Kingdom XP on quest complete
   350→  min_level           INTEGER NOT NULL DEFAULT 1,    -- Locked below this level
   351→  is_active           BOOLEAN NOT NULL DEFAULT TRUE,
   352→  created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
   353→);
   354→
   355→-- User quest instances (one row per quest a user has started)
   356→CREATE TABLE public.user_quests (
   357→  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   358→  user_id               UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
   359→  template_id           UUID NOT NULL REFERENCES public.quest_templates(id),
   360→  status                TEXT NOT NULL DEFAULT 'active'
   361→                          CHECK (status IN ('active', 'completed', 'failed', 'abandoned')),
   362→  objectives_completed  JSONB NOT NULL DEFAULT '[]', -- Array of completed objective IDs
   363→  progress_pct          INTEGER NOT NULL DEFAULT 0 CHECK (progress_pct BETWEEN 0 AND 100),
   364→  started_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
   365→  due_at                TIMESTAMPTZ NOT NULL,
   366→  completed_at          TIMESTAMPTZ,
   367→  failed_at             TIMESTAMPTZ,
   368→  xp_awarded            INTEGER,                     -- Actual XP given (may differ if partial)
   369→  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
   370→);
   371→
   372→-- Index for common queries
   373→CREATE INDEX user_quests_user_status ON public.user_quests(user_id, status);
   374→CREATE INDEX user_quests_due ON public.user_quests(due_at) WHERE status = 'active';
   375→
   376→-- ============================================================
   377→-- KINGDOM SYSTEM
   378→-- ============================================================
   379→
   380→-- Building definitions (seeded by developer)
   381→CREATE TABLE public.buildings (
   382→  id                  TEXT PRIMARY KEY,              -- 'library', 'training_grounds', etc.
   383→  name                TEXT NOT NULL,
   384→  domain              TEXT NOT NULL CHECK (domain IN ('body', 'mind', 'craft', 'command', 'discipline')),
   385→  icon_filename       TEXT NOT NULL,                 -- filename in /public/icons/ e.g. 'book-cover.svg'
   386→  lore_text           TEXT,
   387→  -- Unlock conditions
   388→  unlock_attribute    TEXT NOT NULL,                 -- Column name in profiles, e.g. 'attr_intelligence'
   389→  unlock_threshold    INTEGER NOT NULL,
   390→  -- What the building provides
   391→  quest_unlock_domain TEXT,                          -- Makes quest templates of this domain available
   392→  xp_bonus_pct        INTEGER NOT NULL DEFAULT 0,    -- % XP bonus for quests in unlock domain
   393→  -- Display
   394→  display_order       INTEGER NOT NULL DEFAULT 0,
   395→  phase               INTEGER NOT NULL DEFAULT 1
   396→);
   397→
   398→-- User building status (one row per user per building)
   399→CREATE TABLE public.user_buildings (
   400→  user_id             UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
   401→  building_id         TEXT NOT NULL REFERENCES public.buildings(id),
   402→  status              TEXT NOT NULL DEFAULT 'locked'
   403→                        CHECK (status IN ('locked', 'available', 'built', 'upgrading')),
   404→  built_at            TIMESTAMPTZ,
   405→  PRIMARY KEY (user_id, building_id)
   406→);
   407→
   408→-- ============================================================
   409→-- AI COMPANION
   410→-- ============================================================
   411→
   412→-- Companion message history (max 10 messages per user — prune on insert)
   413→CREATE TABLE public.companion_messages (
   414→  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   415→  user_id             UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
   416→  role                TEXT NOT NULL CHECK (role IN ('user', 'companion')),
   417→  content             TEXT NOT NULL CHECK (char_length(content) BETWEEN 1 AND 2000),
   418→  created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
   419→);
   420→
   421→CREATE INDEX companion_messages_user_time ON public.companion_messages(user_id, created_at DESC);
   422→
   423→-- ============================================================
   424→-- ACHIEVEMENTS
   425→-- ============================================================
   426→
   427→-- Achievement definitions (seeded by developer)
   428→CREATE TABLE public.achievements (
   429→  id                  TEXT PRIMARY KEY,              -- e.g. 'first_quest', 'level_5'
   430→  title               TEXT NOT NULL,
   431→  description         TEXT NOT NULL,
   432→  condition_type      TEXT NOT NULL,                 -- 'quest_count', 'level', 'building', 'domain_quests', 'chat', 'streak', 'all_domains', 'kingdom_level'
   433→  condition_value     INTEGER NOT NULL,              -- The threshold
   434→  condition_domain    TEXT,                          -- For domain-specific achievements
   435→  xp_bonus            INTEGER NOT NULL DEFAULT 0,
   436→  display_order       INTEGER NOT NULL DEFAULT 0
   437→);
   438→
   439→-- User achievement unlocks
   440→CREATE TABLE public.user_achievements (
   441→  user_id             UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
   442→  achievement_id      TEXT NOT NULL REFERENCES public.achievements(id),
   443→  unlocked_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
   444→  PRIMARY KEY (user_id, achievement_id)
   445→);
   446→
   447→-- ============================================================
   448→-- LEADERBOARD (Phase 2 — create table now, populate in Phase 2)
   449→-- ============================================================
   450→
   451→-- Weekly XP snapshot (refreshed every Monday midnight UTC)
   452→CREATE TABLE public.weekly_xp (
   453→  user_id             UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
   454→  house_id            TEXT NOT NULL,
   455→  week_start          DATE NOT NULL,                 -- Monday of the week (ISO week)
   456→  xp_earned           INTEGER NOT NULL DEFAULT 0,
   457→  PRIMARY KEY (user_id, week_start)
   458→);
   459→
   460→CREATE INDEX weekly_xp_house_week ON public.weekly_xp(house_id, week_start, xp_earned DESC);
   461→CREATE INDEX weekly_xp_week ON public.weekly_xp(week_start, xp_earned DESC);
   462→```

   464→### 3.2 Seed Data — Buildings
   465→
   466→```sql
   467→INSERT INTO public.buildings (id, name, domain, icon_filename, lore_text, unlock_attribute, unlock_threshold, quest_unlock_domain, xp_bonus_pct, display_order) VALUES

   697→## 4. Game Systems Specification
   698→
   699→### 4.1 Character System
   700→
   701→#### XP Formula
   702→
   703→```typescript
   704→// lib/xp.ts
   705→
   706→/**
   707→ * XP required to advance FROM level n TO level n+1.
   708→ * Formula: Math.round(250 * Math.pow(level, 1.5))
   709→ * 
   710→ * Level thresholds:
   711→ * Level 1→2:   250 XP  (≈ 1 day for active user)
   712→ * Level 2→3:   707 XP  (≈ 3 days)
   713→ * Level 3→4: 1,299 XP  (≈ 5 days)
   714→ * Level 5→6: 2,795 XP  (≈ 11 days)
   715→ * Level 10→11: 7,906 XP (≈ 32 days)
   716→ * Level 20→21: 22,361 XP (≈ 90 days)
   717→ */
   718→export function xpForLevel(level: number): number {
   719→  return Math.round(250 * Math.pow(level, 1.5))
   720→}
   721→
   722→/**
   723→ * Total cumulative XP required to reach a given level from Level 1.
   724→ */
   725→export function totalXpForLevel(targetLevel: number): number {
   726→  let total = 0
   727→  for (let l = 1; l < targetLevel; l++) {
   728→    total += xpForLevel(l)
   729→  }
   730→  return total
   731→}
   732→
   733→/**
   734→ * Calculate what level a total XP amount corresponds to.
   735→ * Used when awarding large XP bonuses that may trigger multiple level-ups.
   736→ */
   737→export function levelFromTotalXp(totalXp: number): { level: number, xpIntoLevel: number } {
   738→  let level = 1
   739→  let accumulated = 0
   740→  while (true) {
   741→    const next = xpForLevel(level)
   742→    if (accumulated + next > totalXp) {
   743→      return { level, xpIntoLevel: totalXp - accumulated }
   744→    }
   745→    accumulated += next
   746→    level++
   747→    if (level > 999) break // Safety cap
   748→  }
   749→  return { level, xpIntoLevel: 0 }
   750→}
   751→```

   773→#### Attribute Gain per Quest Completion
   774→
   775→Every completed quest awards attribute points. The `primary_attr` and `secondary_attr` columns in `quest_templates` specify which attributes receive points.
   776→
   777→```typescript
   778→// lib/attributes.ts
   779→
   780→// Called inside quest completion handler
   781→export async function awardAttributePoints(
   782→  userId: string,
   783→  primaryAttr: string,  // e.g. 'attr_intelligence'
   784→  secondaryAttr: string // e.g. 'attr_focus'
   785→) {
   786→  const { data: profile } = await supabase
   787→    .from('profiles')
   788→    .select(`${primaryAttr}, ${secondaryAttr}`)
   789→    .eq('id', userId)
   790→    .single()
   791→
   792→  await supabase.from('profiles').update({
   793→    [primaryAttr]: (profile[primaryAttr] || 10) + 2,
   794→    [secondaryAttr]: (profile[secondaryAttr] || 10) + 1,
   795→  }).eq('id', userId)
   796→}
   797→```

   799→#### Discipline Attribute (Streak-Based)
   800→
   801→Discipline increases based on consecutive-day quest completions. It does NOT decrease when a user misses a day (no punishment mechanic).
   802→
   803→```typescript
   804→export async function updateDisciplineStreak(userId: string) {
   805→  // Check if user completed a quest yesterday
   806→  const yesterday = new Date()
   807→  yesterday.setDate(yesterday.getDate() - 1)
   808→  
   809→  const { data: yesterdayQuest } = await supabase
   810→    .from('user_quests')
   811→    .select('id')
   812→    .eq('user_id', userId)
   813→    .eq('status', 'completed')
   814→    .gte('completed_at', yesterday.toISOString().slice(0, 10))
   815→    .lt('completed_at', new Date().toISOString().slice(0, 10))
   816→    .limit(1)
   817→
   818→  if (yesterdayQuest?.length > 0) {
   819→    // Continued streak — gain 1 discipline
   820→    await supabase.from('profiles')
   821→      .update({ attr_discipline: supabase.sql`attr_discipline + 1` })
   822→      .eq('id', userId)
   823→  }
   824→  // No else — discipline never decreases
   825→}
   826→```
   827→
   828→#### Level-Up Sequence
   829→
   830→When a quest completion causes `xp >= xp_to_next`:
   831→
   832→```
   833→1. Calculate new level and remaining XP
   834→2. Update profile: level, xp (remainder), xp_to_next (for new level), xp_total
   835→3. Check if any buildings newly unlock based on updated attributes
   836→4. Check if any achievements trigger based on new level
   837→5. Return { leveledUp: true, newLevel, newTitle, unlockedBuildings, unlockedAchievements }
   838→6. Client receives response and triggers level-up gate animation
   839→```

   880→### 4.2 Quest System
   881→
   882→#### Quest Lifecycle
   883→
   884→```
   885→AVAILABLE → ACTIVE → COMPLETED
   886→                   ↘ FAILED (if due_at passes with incomplete objectives)
   887→                   ↘ ABANDONED (user explicitly abandons)
   888→```
   889→
   890→**State transitions:**
   891→- `available`: Quest exists in `quest_templates` and user hasn't started it. A user can have at most 5 active quests simultaneously.
   892→- `active`: User clicked "Start Quest." Row created in `user_quests` with `status = 'active'` and `due_at = NOW() + duration_days`.
   893→- `completed`: All objectives checked. User clicks "Mark Complete." Backend validates and awards XP.
   894→- `failed`: Server-side cron checks expired quests. If `due_at < NOW()` and `status = 'active'` and `progress_pct < 100`, status → `failed`. A recovery quest is generated.
   895→- `abandoned`: User explicitly chooses to abandon. No XP awarded. Recovery quest offered.
   896→
   897→#### Quest Board Display Rules
   898→
   899→- **Active tab:** All `user_quests` with `status = 'active'` for the current user, sorted by `due_at ASC`.
   900→- **Available tab:** `quest_templates` WHERE `min_level <= profile.level` AND `id NOT IN (recent completions)` AND `class_affinity = [] OR profile.class IN class_affinity`. Ordered by: featured (one epic/rare quest first), then by domain.
   901→- **Completed tab:** All `user_quests` with `status IN ('completed', 'failed', 'abandoned')`, sorted by `completed_at DESC`.
   902→
   903→**Quest count limits:**
   904→- Maximum 5 active quests simultaneously (enforce client-side and server-side)
   905→- No limit on completed quests
   906→
   907→#### Quest Completion Logic (Server-Side — API Route)
   908→
   909→```typescript
   910→// app/api/quests/[id]/complete/route.ts
   911→export async function POST(req: Request, { params }: { params: { id: string } }) {
   912→  const supabase = createServerClient()
   913→  const { data: { user } } = await supabase.auth.getUser()
   914→  if (!user) return Response.json({ error: 'Unauthorized' }, { status: 401 })
   915→
   916→  // 1. Fetch the quest and verify ownership + status
   917→  const { data: quest } = await supabase
   918→    .from('user_quests')
   919→    .select('*, quest_templates(*)')
   920→    .eq('id', params.id)
   921→    .eq('user_id', user.id)
   922→    .single()
   923→
   924→  if (!quest || quest.status !== 'active') {
   925→    return Response.json({ error: 'Quest not found or not active' }, { status: 400 })
   926→  }
   927→
   928→  // 2. Verify all objectives are checked
   929→  const totalObjectives = quest.quest_templates.objectives.length
   930→  const completedCount = quest.objectives_completed.length
   931→  if (completedCount < totalObjectives) {
   932→    return Response.json({ error: 'Not all objectives complete' }, { status: 400 })
   933→  }
   934→
   935→  // 3. Mark quest complete
   936→  await supabase.from('user_quests').update({
   937→    status: 'completed',
   938→    completed_at: new Date().toISOString(),
   939→    xp_awarded: quest.quest_templates.xp_reward,
   940→    progress_pct: 100,
   941→  }).eq('id', params.id)
   942→
   943→  // 4. Award XP and get updated profile
   944→  const template = quest.quest_templates
   945→  const { data: profile } = await supabase.from('profiles').select('*').eq('id', user.id).single()
   946→  
   947→  const newXp = profile.xp + template.xp_reward
   948→  const newXpTotal = profile.xp_total + template.xp_reward
   949→  let newLevel = profile.level
   950→  let currentXp = newXp
   951→  let leveledUp = false
   952→  const levelUps: number[] = []
   953→
   954→  while (currentXp >= xpForLevel(newLevel)) {
   955→    currentXp -= xpForLevel(newLevel)
   956→    newLevel++
   957→    leveledUp = true
   958→    levelUps.push(newLevel)
   959→  }
   960→
   961→  // 5. Award attribute points
   962→  await awardAttributePoints(user.id, template.primary_attr, template.secondary_attr)
   963→
   964→  // 6. Award Kingdom XP
   965→  const newKingdomXp = profile.kingdom_xp + template.kingdom_xp_reward
   966→  // Kingdom level: every 1000 kingdom XP = 1 kingdom level
   967→  const newKingdomLevel = Math.floor(newKingdomXp / 1000) + 1
   968→
   969→  // 7. Update profile
   970→  const { data: updatedProfile } = await supabase.from('profiles').update({
   971→    xp: currentXp,
   972→    xp_total: newXpTotal,
   973→    xp_to_next: xpForLevel(newLevel),
   974→    level: newLevel,
   975→    kingdom_xp: newKingdomXp,
   976→    kingdom_level: newKingdomLevel,
   977→  }).eq('id', user.id).select('*').single()
   978→
   979→  // 8. Check building unlocks
   980→  const unlockedBuildings = await checkBuildingUnlocks(user.id, updatedProfile)
   981→
   982→  // 9. Check achievements
   983→  const unlockedAchievements = await checkAchievements(user.id, updatedProfile)
   984→
   985→  // 10. Update weekly XP (for leaderboard — Phase 2, but write row now)
   986→  await upsertWeeklyXp(user.id, profile.house_id, template.xp_reward)
   987→
   988→  // 11. Update discipline streak
   989→  await updateDisciplineStreak(user.id)
   990→
   991→  return Response.json({
   992→    success: true,
   993→    xpAwarded: template.xp_reward,
   994→    leveledUp,
   995→    newLevel,
   996→    newTitle: leveledUp ? getTitleForLevel(newLevel) : null,
   997→    unlockedBuildings,
   998→    unlockedAchievements,
   999→  })
  1000→}
  1001→```

  1046→#### Building Unlock State Machine
  1047→
  1048→```
  1049→LOCKED → AVAILABLE → BUILT
  1050→         (conditions met)   (user clicks "Build")

  1052→
  1053→Unlocking happens automatically server-side when attribute thresholds are met after a quest completion. "Building" (transitioning from AVAILABLE to BUILT) is a user-triggered action — no cost, no timer, no currency. The unlock IS the reward.

  1056→

  1058→// Kingdom XP accumulates separately from character XP

  1061→// Kingdom Level unlocks higher-tier buildings (Phase 2+)

  1064→  return Math.floor(kingdomXp / 1000) + 1

  1089→  const { data: lockedBuildings } = await supabase

  1096→
  1097→  for (const ub of lockedBuildings ?? []) {
  1098→    const { unlock_attribute, unlock_threshold } = ub.buildings

  1100→

  1105→        .eq('building_id', ub.building_id)

  1133→```typescript
  1134→// lib/companion.ts
  1135→
  1136→const RULE_RESPONSES: Record<string, string> = {
  1137→  'quest':      'Your board has {activeCount} active trials. Your {topDomain} work is strongest. Want a new mission aligned with that path?',
  1138→  'level':      'You are {xpToNext} echoes from Fragment {nextLevel}. Two quests from your board would close that distance.',

  1140→  'tired':      'Rest today. Your active trials will not expire for {hoursUntilDue} hours. Return when the stillness passes.',
  1141→  'help':       'I can generate a trial, read your progress, or suggest a direction. Name your need.',
  1142→  'kingdom':    'Your realm holds {builtCount} built monuments and {availableCount} ready to manifest. {buildingName} awaits your command.',
  1143→  'achieve':    'You have claimed {achievementCount} milestones. Your nearest is "{nextAchievement}" — {nextAchievementProgress}.',
  1144→  'hello':      'You returned, {characterName}. The void has been watching. What do we build today?',
  1145→  'hi':         'You returned, {characterName}. The void has been watching. What do we build today?',
  1146→  'who are you': 'I am {companionName}, your architect of intent. I observe your progress, generate your trials, and keep you honest. Ask me anything.',
  1147→}
  1148→
  1149→function fillTemplate(template: string, ctx: UserContext): string {
  1150→  return template
  1151→    .replace('{characterName}',         ctx.characterName)
  1152→    .replace('{companionName}',         ctx.companionName)
  1153→    .replace('{activeCount}',           String(ctx.activeQuestCount))
  1154→    .replace('{topDomain}',             ctx.topDomain)
  1155→    .replace('{xpToNext}',              String(ctx.xpToNext))
  1156→    .replace('{nextLevel}',             String(ctx.level + 1))
  1157→    .replace('{hoursUntilDue}',         String(ctx.hoursUntilDue))
  1158→    .replace('{builtCount}',            String(ctx.builtBuildingCount))
  1159→    .replace('{availableCount}',        String(ctx.availableBuildingCount))
  1160→    .replace('{buildingName}',          ctx.nextAvailableBuilding ?? 'a monument')
  1161→    .replace('{achievementCount}',      String(ctx.achievementCount))
  1162→    .replace('{nextAchievement}',       ctx.nextAchievement ?? 'First Blood')
  1163→    .replace('{nextAchievementProgress}', ctx.nextAchievementProgress ?? 'begin')
  1164→}
  1165→
  1166→export async function getCompanionResponse(
  1167→  message: string,
  1168→  userContext: UserContext,
  1169→  conversationHistory: CompanionMessage[]
  1170→): Promise<string> {
  1171→  const lower = message.toLowerCase().trim()
  1172→
  1173→  // Rule-based check — no API cost
  1174→  for (const [keyword, template] of Object.entries(RULE_RESPONSES)) {
  1175→    if (lower.includes(keyword)) {
  1176→      return fillTemplate(template, userContext)
  1177→    }
  1178→  }
  1179→
  1180→  // Gemini fallback
  1181→  try {
  1182→    return await callGemini(message, userContext, conversationHistory)
  1183→  } catch (error) {
  1184→    // Groq fallback
  1185→    try {
  1186→      return await callGroq(message, userContext, conversationHistory)
  1187→    } catch {
  1188→      // Last resort: rule-based generic response
  1189→      return fillTemplate(RULE_RESPONSES['help'], userContext)
  1190→    }
  1191→  }
  1192→}

  1271→### 4.5 Achievement System
  1272→
  1273→Achievement evaluation runs server-side, triggered after every quest completion and level-up.
  1274→
  1275→```typescript
  1276→// lib/achievements.ts
  1277→
  1278→export async function checkAchievements(
  1279→  userId: string,
  1280→  profile: Profile
  1281→): Promise<string[]> {
  1282→  // Get all achievements not yet earned by this user
  1283→  const { data: unearned } = await supabase
  1284→    .from('achievements')
  1285→    .select('*')
  1286→    .not('id', 'in', 
  1287→      `(SELECT achievement_id FROM user_achievements WHERE user_id = '${userId}')`
  1288→    )
  1289→
  1290→  const unlocked: string[] = []
  1291→
  1292→  for (const achievement of unearned ?? []) {
  1293→    let earned = false
  1294→
  1295→    switch (achievement.condition_type) {
  1296→      case 'quest_count': {
  1297→        const { count } = await supabase
  1298→          .from('user_quests')
  1299→          .select('*', { count: 'exact', head: true })
  1300→          .eq('user_id', userId)
  1301→          .eq('status', 'completed')
  1302→        earned = (count ?? 0) >= achievement.condition_value
  1303→        break
  1304→      }
  1305→      case 'level':
  1306→        earned = profile.level >= achievement.condition_value
  1307→        break
  1308→      case 'building': {
  1309→        const { count } = await supabase
  1310→          .from('user_buildings')
  1311→          .select('*', { count: 'exact', head: true })
  1312→          .eq('user_id', userId)
  1313→          .eq('status', 'built')
  1314→        earned = (count ?? 0) >= achievement.condition_value
  1315→        break
  1316→      }
  1317→      case 'domain_quests': {
  1318→        const { count } = await supabase
  1319→          .from('user_quests')
  1320→          .select('*, quest_templates!inner(domain)', { count: 'exact', head: true })
  1321→          .eq('user_id', userId)
  1322→          .eq('status', 'completed')
  1323→          .eq('quest_templates.domain', achievement.condition_domain)
  1324→        earned = (count ?? 0) >= achievement.condition_value
  1325→        break
  1326→      }
  1327→      case 'chat': {
  1328→        const { count } = await supabase
  1329→          .from('companion_messages')
  1330→          .select('*', { count: 'exact', head: true })
  1331→          .eq('user_id', userId)
  1332→          .eq('role', 'user')
  1333→        earned = (count ?? 0) >= achievement.condition_value
  1334→        break
  1335→      }
  1336→      case 'kingdom_level':
  1337→        earned = profile.kingdom_level >= achievement.condition_value
  1338→        break
  1339→      case 'all_domains': {
  1340→        const domains = ['body', 'mind', 'craft', 'command']
  1341→        const checks = await Promise.all(domains.map(async domain => {
  1342→          const { count } = await supabase
  1343→            .from('user_quests')
  1344→            .select('*, quest_templates!inner(domain)', { count: 'exact', head: true })
  1345→            .eq('user_id', userId)
  1346→            .eq('status', 'completed')
  1347→            .eq('quest_templates.domain', domain)
  1348→          return (count ?? 0) > 0
  1349→        }))
  1350→        earned = checks.every(Boolean)
  1351→        break
  1352→      }
  1353→      case 'streak':
  1354→        // 7-day streak check
  1355→        const sevenDaysAgo = new Date()
  1356→        sevenDaysAgo.setDate(sevenDaysAgo.getDate() - 7)
  1357→        // Count distinct days with completions in the last 7 days
  1358→        const { data: recentCompletions } = await supabase
  1359→          .from('user_quests')
  1360→          .select('completed_at')
  1361→          .eq('user_id', userId)
  1362→          .eq('status', 'completed')
  1363→          .gte('completed_at', sevenDaysAgo.toISOString())
  1364→        const distinctDays = new Set(
  1365→          recentCompletions?.map(q => q.completed_at?.slice(0, 10)) ?? []
  1366→        )
  1367→        earned = distinctDays.size >= 7
  1368→        break
  1369→    }
  1370→
  1371→    if (earned) {
  1372→      await supabase.from('user_achievements').insert({
  1373→        user_id: userId,
  1374→        achievement_id: achievement.id,
  1375→      })
  1376→      // Award XP bonus
  1377→      if (achievement.xp_bonus > 0) {
  1378→        await supabase.from('profiles').update({
  1379→          xp: supabase.sql`xp + ${achievement.xp_bonus}`,
  1380→          xp_total: supabase.sql`xp_total + ${achievement.xp_bonus}`,
  1381→        }).eq('id', userId)
  1382→      }
  1383→      unlocked.push(achievement.id)
  1384→    }
  1385→  }
  1386→
  1387→  return unlocked
  1388→}
  1389→```

  1640→## 6. State Management & Logic Flow
  1641→
  1642→### 6.1 Client-Side State Architecture
  1643→
  1644→No external state library. Use React Context for shared state and `useState`/`useReducer` for local state.
  1645→
  1646→```typescript
  1647→// contexts/UserContext.tsx
  1648→interface UserContextType {
  1649→  profile: Profile | null
  1650→  loading: boolean
  1651→  refetchProfile: () => Promise<void>
  1652→}
  1653→
  1654→export const UserContext = createContext<UserContextType>({ ... })
  1655→
  1656→// Wrap app in UserProvider in app/layout.tsx
  1657→// All pages that need profile data consume useUser() hook
  1658→export function useUser() {
  1659→  return useContext(UserContext)
  1660→}
  1661→```
  1662→
  1663→**What lives in context:** The current user's profile (character stats, level, XP). Refreshed after any mutation (quest complete, level-up, building build).
  1664→
  1665→**What lives in local state:** Page-specific data (quest list, companion history, building grid). Each route fetches its own data on mount via `useEffect` or Server Components.
  1666→
  1667→### 6.2 Server vs Client Data Fetching
  1668→
  1669→| Data | Fetch Method | When |
  1670→|---|---|---|
  1671→| User profile | Client-side (context, via Supabase JS client) | On auth, after mutations |
  1672→| Quest templates (available) | Server Component | On `/quests` page load |
  1673→| Active user quests | Client-side | On `/quests` and `/dashboard` |
  1674→| Companion history | Client-side | On `/companion` mount |
  1675→| Kingdom buildings | Server Component | On `/kingdom` page load |
  1676→| Leaderboard | Server Component | On `/leaderboard` page load |
  1677→
  1678→### 6.3 Quest Completion State Flow
  1679→
  1680→```
  1681→User clicks "Mark Complete" button
  1682→  → Client: disable button, show spinner
  1683→  → POST /api/quests/:id/complete
  1684→  → Server: validate, award XP, check level-up, check buildings, check achievements
  1685→  → Server: return { xpAwarded, leveledUp, newLevel, unlockedBuildings, unlockedAchievements }
  1686→  → Client: receive response
  1687→    → Fire XP toast (+N XP)
  1688→    → Animate XP bar fill
  1689→    → If leveledUp: show level-up gate (full screen overlay)
  1690→    → If unlockedBuildings: show building unlock toast
  1691→    → If unlockedAchievements: show achievement toast (queue if multiple)
  1692→    → Refetch profile (to update sidebar XP bar and level display)
  1693→    → Remove completed quest from active list
  1694→```
  1695→
  1696→### 6.4 Offline Behavior
  1697→
  1698→Sovereign does NOT support offline mode. All actions require an internet connection.
  1699→
  1700→If a user is offline:
  1701→- Forms display: "Could not connect. Please check your internet connection."
  1702→- Quest completion shows error toast: "Connection lost. Your progress is saved locally — try again when online."
  1703→- Do NOT queue actions for sync. The app simply does nothing until reconnected.
  1704→

  1808→### 7.5 Toast Queue Management
  1809→
  1810→```typescript
  1811→// lib/toasts.ts
  1812→// Toasts must not overlap. Queue them with delays.
  1813→
  1814→interface ToastItem {
  1815→  type: 'xp' | 'quest-complete' | 'building-unlock' | 'achievement'
  1816→  message: string
  1817→  delay: number // ms from trigger
  1818→}
  1819→
  1820→export function buildToastQueue(completionResult: QuestCompletionResult): ToastItem[] {
  1821→  const queue: ToastItem[] = []
  1822→  let delay = 0
  1823→
  1824→  // XP toast appears immediately after completion
  1825→  queue.push({ type: 'xp', message: `+${completionResult.xpAwarded} XP`, delay })
  1826→  delay += 500 // 500ms after XP toast
  1827→
  1828→  // Quest complete toast
  1829→  queue.push({ type: 'quest-complete', message: 'Trial complete.', delay })
  1830→  delay += 3500 // 3.5s (3s display + 500ms gap)
  1831→
  1832→  // Building unlock toasts (if any)
  1833→  for (const building of completionResult.unlockedBuildings) {
  1834→    queue.push({ type: 'building-unlock', message: `${building} unlocked.`, delay })
  1835→    delay += 3500
  1836→  }
  1837→
  1838→  // Achievement toasts (if any)
  1839→  for (const achievement of completionResult.unlockedAchievements) {
  1840→    queue.push({ type: 'achievement', message: `Achievement: ${achievement}`, delay })
  1841→    delay += 3500
  1842→  }
  1843→
  1844→  return queue
  1845→}
  1846→```
