    29→    29→## 1. Product Definition & Scope
    30→    30→
    31→    31→### 1.1 What This Product Is
    32→    32→
    33→    33→Sovereign is a browser-based web application that turns real-life self-improvement into an RPG kingdom-building game. Users complete real-world actions — studying, exercising, building skills, leading — and those actions award XP, level up their character, grow attributes, and unlock buildings in a persistent kingdom.
    34→    34→
    35→    35→It is NOT a to-do list. It is NOT a habit tracker with cosmetic decoration. It is an identity system that happens to use quests as its input mechanism.
    36→    36→
    37→    37→### 1.2 What This Product Is Not
    38→    38→
    39→    39→- Not a downloadable game (no Unity, Godot, or game engine)
    40→    40→- Not a mobile app (no React Native or Flutter)
    41→    41→- Not a social media platform
    42→    42→- Not a habit tracker
    43→    43→- Not a task manager
    44→    44→- Not a dashboard or analytics tool
    45→    45→
    46→    46→The kingdom map is a styled CSS div layout. The XP bar is a CSS transition. The AI companion is a fetch call. Everything runs in a browser via URL.
    47→    47→
    48→    48→### 1.3 Target Platform
    49→    49→
    50→    50→**FIXED:** Next.js 14 web application, deployed on Vercel, accessed via browser at a custom domain.
    51→    51→
    52→    52→Same technical form factor as Linear, Notion, and Habitica — all of which are React/Next.js web apps with game-like UI.
    53→    53→
    54→    54→### 1.4 Target User
    55→    55→
    56→    56→Primary demographic: Ambitious males aged 18–24 who consume self-improvement content (YouTube, Reddit, podcasts), are familiar with gaming progression systems (anime leveling aesthetics, RPG stat screens), and feel productivity apps are either too boring or too childish.
    57→    57→
    58→    58→**IMPORTANT — This demographic is a hypothesis, not a fact.** The 20-person test cohort in Phase 2 Week 14 must validate this assumption. The product should be built as though the assumption is true, but the retention metrics will reveal whether it holds.
    59→    59→
    60→    60→Evidence supporting the hypothesis:
    61→    61→- HabitForge (Solo Leveling-inspired habit tracker) gained a user base in this demographic with anime-leveling aesthetics
    62→    62→- Reddit communities such as r/selfimprovement, r/nosurf, and r/getdisciplined skew young male
    63→    63→- LifeUp has 280K+ downloads with user reviews indicating gaming-oriented self-improvement seekers
    64→    64→
    65→    65→Evidence against: LifeUp's top complaint is that it "lacks a real game." Habitica lost its community spaces in 2023 and its guilds in 2023 due to moderation issues. This means the category has a demand problem (users want a more serious experience) and a supply gap.
    66→    66→
    67→    67→### 1.5 Core Loop
    68→    68→
    69→    69→**Single session (under 5 minutes for casual engagement):**
    70→    70→
    71→    71→```
    72→    72→Open app
    73→    73→→ Check active quests on Dashboard
    74→    74→→ Mark objective(s) complete on one quest
    75→    75→→ See XP animation fill the bar
    76→    76→→ [Optional] Check kingdom for any building unlocks
    77→    77→→ [Optional] Message companion
    78→    78→→ Close app
    79→    79→```
    80→    80→
    81→    81→**Daily loop (5–15 minutes for power users):**
    82→    82→
    83→    83→```
    84→    84→Morning: Check companion for daily quest recommendation
    85→    85→→ Browse Quest Board for available quests
    86→    86→→ Start 1–2 new quests
    87→    87→→ Mark progress on active quests
    88→    88→
    89→    89→Evening: Mark completions
    90→    90→→ XP award and animation
    91→    91→→ Level-up gate if threshold crossed
    92→    92→→ Building unlock check
    93→    93→→ Achievement check
    94→    94→→ Check guild weekly progress
    95→    95→```
    96→    96→
    97→    97→**Weekly loop:**
    98→    98→
    99→    99→```
   100→   100→Monday: House weekly collective XP goal resets
   101→   101→→ New featured quest appears on Quest Board
   102→   102→→ Review progress toward Kingdom Level milestone
   103→   103→
   104→   104→Friday: Check House leaderboard position
   105→   105→→ Final push on active quests before week resets
   106→   106→```
   107→   107→
   108→   108→### 1.6 Session Length Targets
   109→   109→
   110→   110→- Minimum meaningful session: 30 seconds (mark one objective complete)
   111→   111→- Casual session: 2–5 minutes (check dashboard, update quest, view kingdom)
   112→   112→- Power user session: 10–20 minutes (quest board review, companion chat, character review)
   113→   113→- Onboarding session: 90 seconds maximum (3 steps, one page)
   114→   114→
   115→   115→### 1.7 Win State and Endgame
   116→   116→
   117→   117→There is no win condition. Sovereign is an ongoing identity game.
   118→   118→
   119→   119→Milestones that replace a "win state":
   120→   120→- Reach Level 10 (Journeyman status)
   121→   121→- Build all 6 Phase 1 kingdom buildings
   122→   122→- Complete quests in every domain (Renaissance achievement)
   123→   123→- Reach Kingdom Level 5 (Phase 2)
   124→   124→- Reach the top 10 of the House weekly leaderboard (Phase 2)
   125→   125→
   126→   126→The "endgame" is a high-level character with a complete kingdom, a recognizable title in the leaderboard, and a reputation in their House. Phase 3 will introduce prestige mechanics (class reset with bonuses) for users who have reached Level 50+.

   130→   130→## 2. Technical Architecture
   131→   131→
   132→   132→### 2.1 Framework and Engine
   133→   133→
   134→   134→**FIXED: Next.js 14 with App Router. No other framework is acceptable.**
   135→   135→
   136→   136→Justification:
   137→   137→- Single repo for frontend, API routes, and server components
   138→   138→- App Router enables server-side data fetching without separate API layer for most reads
   139→   139→- Vercel deploys Next.js natively with zero configuration
   140→   140→- shadcn/ui is built for Next.js + Tailwind
   141→   141→- Supabase has a first-class Next.js integration package
   142→   142→
   143→   143→Do NOT use: Vite+React (no SSR, no API routes), Remix (less ecosystem support with Supabase), SvelteKit (unfamiliar tech stack risk).
   144→   144→
   145→   145→### 2.2 Frontend Stack

   147→   147→| Layer | Technology | Version | Purpose |
   148→   148→|---|---|---|---|
   149→   149→| Framework | Next.js | 14.2.x | App Router, SSR, API Routes |
   150→   150→| Language | TypeScript | 5.x | Type safety |
   151→   151→| Styling | Tailwind CSS | 3.4.x | Utility-first CSS |

   189→   189→### 2.4 AI Integration
   190→   190→
   191→   191→**FIXED: Gemini 3 Flash as primary model. Gemini 2.0 Flash was deprecated and shut down June 1, 2026 — do not reference it anywhere in code.**
   192→   192→
   193→   193→| Service | Model | Free Limit | RPM | RPD | Use |
   194→   194→|---|---|---|---|---|---|
   195→   195→| Google AI Studio | gemini-3-flash (or `gemini-2.5-flash` if 3 unavailable) | Free, no credit card | 10 | 1,500 | Companion chat, quest lore personalization |
   196→   196→| Groq | llama-3.1-8b-instant | Free, no credit card | varies | 14,400 | Fallback when Gemini unavailable |
   197→   197→
   198→   198→**API key management:**
   199→   199→- `GEMINI_API_KEY` — server-side only, in `.env.local`, never in client code
   200→   200→- Never prefix with `NEXT_PUBLIC_` — would expose the key in browser
   201→   201→- All Gemini calls go through Next.js API Route Handlers in `/app/api/`
   202→   202→
   203→   203→**AI strategy:** Template-based + light AI personalization. The AI does NOT generate quest content. It only:
   204→   204→1. Personalizes lore text for existing quest templates (1 API call per quest generation, ~20 words output)
   205→   205→2. Responds to companion chat messages (1 API call per message, 200 token max output)
   206→   206→
   207→   207→At 50 DAU × 3 AI calls/day = 150 calls/day. Well within 1,500 RPD free limit.
   208→   208→
   209→   209→**Rate limit handling:**
   210→   210→```typescript
   211→   211→// lib/gemini.ts — exponential backoff on 429
   212→   212→async function callGemini(prompt: string, maxRetries = 3): Promise<string> {
   213→   213→  for (let attempt = 0; attempt < maxRetries; attempt++) {
   214→   214→    try {
   215→   215→      const response = await fetch(GEMINI_API_URL, { ... })
   216→   216→      if (response.status === 429) {
   217→   217→        await sleep(Math.pow(2, attempt) * 1000) // 1s, 2s, 4s
   218→   218→        continue
   219→   219→      }
   220→   220→      return await parseResponse(response)
   221→   221→    } catch (e) {
   222→   222→      if (attempt === maxRetries - 1) throw e
   223→   223→    }
   224→   224→  }
   225→   225→  return RULE_BASED_FALLBACK // If all retries fail, use rule-based response
   226→   226→}
   227→   227→```
   228→   228→
   229→   229→**Fallback to Groq:** If Gemini returns 429 after 3 retries, call Groq's llama-3.1-8b-instant with the same prompt. If Groq also fails, return a rule-based response from the RULE_RESPONSES dictionary (no API call).

   282→   282→## 3. Complete Data Model
   283→   283→
   284→   284→### 3.1 Database Tables — Phase 1
   285→   285→
   286→   286→All tables are in the `public` schema. All tables have Row Level Security enabled.
   287→   287→
   288→   288→```sql
   289→   289→-- ============================================================
   290→   290→-- CORE USER DATA
   291→   291→-- ============================================================
   292→   292→
   293→   293→-- User profiles (extends Supabase auth.users)
   294→   294→CREATE TABLE public.profiles (
   295→   295→  id                  UUID REFERENCES auth.users(id) ON DELETE CASCADE PRIMARY KEY,
   296→   296→  character_name      TEXT NOT NULL CHECK (char_length(character_name) BETWEEN 2 AND 30),
   297→   297→  kingdom_name        TEXT NOT NULL CHECK (char_length(kingdom_name) BETWEEN 2 AND 40),
   298→   298→  companion_name      TEXT NOT NULL DEFAULT 'Aegis' CHECK (char_length(companion_name) BETWEEN 2 AND 20),
   299→   299→  class               TEXT NOT NULL CHECK (class IN ('scholar', 'warrior', 'builder', 'commander')),
   300→   300→  house_id            TEXT NOT NULL DEFAULT 'zenith' CHECK (house_id IN ('ash', 'zenith', 'forge', 'crown')),
   301→   301→  house_changed_at    TIMESTAMPTZ,                  -- Enforce once-per-week change limit
   302→   302→  level               INTEGER NOT NULL DEFAULT 1 CHECK (level >= 1),
   303→   303→  xp                  INTEGER NOT NULL DEFAULT 0 CHECK (xp >= 0),
   304→   304→  xp_to_next          INTEGER NOT NULL DEFAULT 300, -- XP needed for current level → next level
   305→   305→  xp_total            INTEGER NOT NULL DEFAULT 0,   -- Lifetime XP (never resets on level-up)
   306→   306→  kingdom_level       INTEGER NOT NULL DEFAULT 1 CHECK (kingdom_level >= 1),
   307→   307→  kingdom_xp          INTEGER NOT NULL DEFAULT 0,   -- Separate from character XP
   308→   308→  -- Attributes (all start at 10, grow from quest completion)
   309→   309→  attr_strength       INTEGER NOT NULL DEFAULT 10 CHECK (attr_strength >= 0),
   310→   310→  attr_vitality       INTEGER NOT NULL DEFAULT 10 CHECK (attr_vitality >= 0),
   311→   311→  attr_intelligence   INTEGER NOT NULL DEFAULT 10 CHECK (attr_intelligence >= 0),
   312→   312→  attr_focus          INTEGER NOT NULL DEFAULT 10 CHECK (attr_focus >= 0),
   313→   313→  attr_technical      INTEGER NOT NULL DEFAULT 10 CHECK (attr_technical >= 0),
   314→   314→  attr_creativity     INTEGER NOT NULL DEFAULT 10 CHECK (attr_creativity >= 0),
   315→   315→  attr_leadership     INTEGER NOT NULL DEFAULT 10 CHECK (attr_leadership >= 0),
   316→   316→  attr_charisma       INTEGER NOT NULL DEFAULT 10 CHECK (attr_charisma >= 0),
   317→   317→  attr_discipline     INTEGER NOT NULL DEFAULT 10 CHECK (attr_discipline >= 0), -- Streak-based
   318→   318→  -- Meta
   319→   319→  onboarding_complete BOOLEAN NOT NULL DEFAULT FALSE,
   320→   320→  created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
   321→   321→  updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
   322→   322→);
   323→   323→
   324→   324→-- Trigger: auto-update updated_at
   325→   325→CREATE OR REPLACE FUNCTION update_updated_at()
   326→   326→RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = NOW(); RETURN NEW; END; $$ LANGUAGE plpgsql;
   327→   327→CREATE TRIGGER profiles_updated_at BEFORE UPDATE ON public.profiles
   328→   328→  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
   329→   329→
   330→   330→-- ============================================================
   331→   331→-- QUEST SYSTEM
   332→   332→-- ============================================================
   333→   333→
   334→   334→-- Quest templates (seeded by developer, not user-generated at Phase 1)
   335→   335→CREATE TABLE public.quest_templates (
   336→   336→  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   337→   337→  title               TEXT NOT NULL CHECK (char_length(title) BETWEEN 5 AND 80),
   338→   338→  description         TEXT NOT NULL CHECK (char_length(description) BETWEEN 10 AND 200),
   339→   339→  domain              TEXT NOT NULL CHECK (domain IN ('body', 'mind', 'craft', 'command')),
   340→   340→  difficulty          TEXT NOT NULL DEFAULT 'medium' CHECK (difficulty IN ('easy', 'medium', 'hard', 'epic')),
   341→   341→  class_affinity      TEXT[] DEFAULT '{}',           -- Empty array = available to all classes
   342→   342→  xp_reward           INTEGER NOT NULL CHECK (xp_reward > 0),
   343→   343→  rarity              TEXT NOT NULL DEFAULT 'common' CHECK (rarity IN ('common', 'uncommon', 'rare', 'epic')),
   344→   344→  duration_days       INTEGER NOT NULL DEFAULT 7,
   345→   345→  objectives          JSONB NOT NULL DEFAULT '[]',   -- [{id: UUID, text: string, order: number}]
   346→   346→  -- Attribute rewards on completion
   347→   347→  primary_attr        TEXT NOT NULL,                 -- Which attribute gets +2
   348→   348→  secondary_attr      TEXT NOT NULL,                 -- Which attribute gets +1
   349→   349→  kingdom_xp_reward   INTEGER NOT NULL DEFAULT 50,   -- Kingdom XP on quest complete
   350→   350→  min_level           INTEGER NOT NULL DEFAULT 1,    -- Locked below this level
   351→   351→  is_active           BOOLEAN NOT NULL DEFAULT TRUE,
   352→   352→  created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
   353→   353→);
   354→   354→
   355→   355→-- User quest instances (one row per quest a user has started)
   356→   356→CREATE TABLE public.user_quests (
   357→   357→  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   358→   358→  user_id               UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
   359→   359→  template_id           UUID NOT NULL REFERENCES public.quest_templates(id),
   360→   360→  status                TEXT NOT NULL DEFAULT 'active'
   361→   361→                          CHECK (status IN ('active', 'completed', 'failed', 'abandoned')),
   362→   362→  objectives_completed  JSONB NOT NULL DEFAULT '[]', -- Array of completed objective IDs
   363→   363→  progress_pct          INTEGER NOT NULL DEFAULT 0 CHECK (progress_pct BETWEEN 0 AND 100),
   364→   364→  started_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
   365→   365→  due_at                TIMESTAMPTZ NOT NULL,
   366→   366→  completed_at          TIMESTAMPTZ,
   367→   367→  failed_at             TIMESTAMPTZ,
   368→   368→  xp_awarded            INTEGER,                     -- Actual XP given (may differ if partial)
   369→   369→  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
   370→   370→);
   371→   371→
   372→   372→-- Index for common queries
   373→   373→CREATE INDEX user_quests_user_status ON public.user_quests(user_id, status);
   374→   374→CREATE INDEX user_quests_due ON public.user_quests(due_at) WHERE status = 'active';
   375→   375→
   376→   376→-- ============================================================
   377→   377→-- KINGDOM SYSTEM
   378→   378→-- ============================================================
   379→   379→
   380→   380→-- Building definitions (seeded by developer)
   381→   381→CREATE TABLE public.buildings (
   382→   382→  id                  TEXT PRIMARY KEY,              -- 'library', 'training_grounds', etc.
   383→   383→  name                TEXT NOT NULL,
   384→   384→  domain              TEXT NOT NULL CHECK (domain IN ('body', 'mind', 'craft', 'command', 'discipline')),
   385→   385→  icon_filename       TEXT NOT NULL,                 -- filename in /public/icons/ e.g. 'book-cover.svg'
   386→   386→  lore_text           TEXT,
   387→   387→  -- Unlock conditions
   388→   388→  unlock_attribute    TEXT NOT NULL,                 -- Column name in profiles, e.g. 'attr_intelligence'
   389→   389→  unlock_threshold    INTEGER NOT NULL,
   390→   390→  -- What the building provides
   391→   391→  quest_unlock_domain TEXT,                          -- Makes quest templates of this domain available
   392→   392→  xp_bonus_pct        INTEGER NOT NULL DEFAULT 0,    -- % XP bonus for quests in unlock domain
   393→   393→  -- Display
   394→   394→  display_order       INTEGER NOT NULL DEFAULT 0,
   395→   395→  phase               INTEGER NOT NULL DEFAULT 1
   396→   396→);
   397→   397→
   398→   398→-- User building status (one row per user per building)
